# Robot Grasp Detection

Проект по производственной, проектно-технологической практике.

Тема: разработка алгоритма детекции объектов сложной формы для адаптивного захватного устройства робота.

Цель проекта - реализовать программный пайплайн, который находит объект на изображении, выделяет область интереса, уточняет форму объекта средствами OpenCV и рассчитывает параметры, которые могут быть использованы для предполагаемого захвата.

## Основная идея

Алгоритм состоит из двух основных частей:

1. YOLO определяет объект на изображении и возвращает bounding box.
2. OpenCV обрабатывает найденную область интереса, выделяет контур объекта и рассчитывает параметры захвата.

Общий пайплайн:

```text
image -> YOLO -> ROI -> OpenCV -> contour -> grasp line
```

На выходе формируются:

* класс объекта;
* bounding box;
* ROI;
* бинарная маска;
* контур объекта;
* центр объекта;
* размеры объекта;
* площадь контура;
* угол ориентации;
* предполагаемая линия захвата.

## Этап 3. Выделение ROI и обработка OpenCV

После YOLO-детекции из изображения вырезается ROI - область интереса. Это позволяет обрабатывать не всё изображение, а только участок, где находится объект.

Основные шаги обработки ROI:

1. перевод изображения в LAB;
2. улучшение яркостного канала через CLAHE;
3. размытие;
4. бинаризация;
5. морфологические операции;
6. поиск контуров;
7. выбор основного контура по площади.

Для разных классов использовались разные параметры обработки:

* для `cell` - adaptive threshold;
* для `sensor` - Otsu threshold.

## Этап 4. Расчёт параметров объекта

После нахождения контура рассчитываются параметры объекта:

* площадь контура;
* центр объекта;
* минимальный ограничивающий прямоугольник;
* ширина и высота;
* длинная и короткая стороны;
* угол ориентации;
* предполагаемая ширина захвата;
* предполагаемая линия захвата.

Для оценки ориентации используется `cv2.minAreaRect`.

Пример расчёта параметров:

```python
best_cont = max(conts, key=cv2.contourArea)
s_cont = cv2.contourArea(best_cont)

approx = cv2.approxPolyDP(
    best_cont,
    0.01 * cv2.arcLength(best_cont, True),
    True
)

hull = cv2.convexHull(approx)
rect = cv2.minAreaRect(hull)

(c_x, c_y), (w, h), ang = rect

long_side = max(w, h)
short_side = min(w, h)

grasp_ang = (ang if w < h else ang + 90) % 180
grasp_width = short_side * k
```

Финальный результат содержит:

* зелёный контур объекта;
* красный минимальный прямоугольник;
* синюю точку центра;
* голубую линию предполагаемого захвата.

## Этап 5. Квантизация и оптимизация модели

Была выполнена попытка оптимизации модели:

| Файл                   | Формат    | Назначение                                |
| ---------------------- | --------- | ----------------------------------------- |
| `yolo_model.pt`        | PyTorch   | исходная обученная модель                 |
| `yolo_model.onnx`      | ONNX FP32 | экспортированная модель                   |
| `yolo_model_int8.onnx` | ONNX INT8 | экспериментальная квантизированная версия |

INT8-модель теоретически должна быть легче, так как часть значений переводится в 8-битный целочисленный формат. Это может уменьшить размер модели и ускорить инференс на устройствах с хорошей поддержкой INT8.

Но в текущей реализации INT8-квантизация оказалась нерабочей для данного пайплайна:

* размер модели уменьшился;
* метрики детекции стали нулевыми;
* `mAP50 = 0.0`;
* `mAP50-95 = 0.0`.

Причины:

* потеря точности при квантизации;
* возможная несовместимость INT8 ONNX-модели с backend инференса;
* падение confidence после перевода весов и активаций в INT8.

Поэтому основной рабочей моделью осталась:

```text
models/yolo_model.pt
```

ONNX FP32 рассматривалась как рабочий вариант экспорта без квантизации.

## Этап 6. Тестирование и анализ ошибок

Финальное тестирование выполнялось на `dataset/test/images`.

Для каждого изображения выполнялся полный пайплайн:

```text
image -> YOLO -> ROI -> OpenCV -> contour -> grasp line
```

Результаты тестирования:

| Категория                                              | Количество |
| ------------------------------------------------------ | ---------: |
| Успешная обработка                                     |         18 |
| Ложные срабатывания                                    |          0 |
| Пропуски объектов или отсутствие финального результата |          9 |
| Неточное выделение контура                             |         17 |
| Всего изображений в test                               |         44 |

Из 44 изображений тестовой выборки для 35 изображений был получен финальный результат обработки. Из них 18 результатов были оценены как успешные, а в 17 случаях наблюдалось неточное выделение контура. Для 9 изображений финальный результат не был сформирован.

Основные ошибки были связаны не с первичной YOLO-детекцией, а с этапом уточнения формы средствами OpenCV.

### Успешная обработка

Успешной считалась обработка, при которой:

* объект найден YOLO-моделью;
* ROI содержит объект;
* контур выделен корректно;
* линия захвата соответствует ориентации объекта.



### Пропуски объектов или отсутствие финального результата

К этой категории относятся случаи, когда объект не был найден моделью YOLO либо дальнейшая обработка ROI не позволила получить подходящий контур.

Возможные причины:

* объект плохо виден;
* объект частично перекрыт;
* недостаточное освещение;
* редкий ракурс объекта;
* confidence ниже заданного порога;
* плохой фон или качество ROI.


### Неточное выделение контура

Иногда YOLO корректно находит объект, но OpenCV выделяет контур неточно.

Возможные причины:

* пересвет;
* тени;
* низкий контраст между объектом и фоном;
* шум после бинаризации;
* сложная форма объекта.



### Нестабильный угол ориентации

Угол ориентации может быть нестабильным, если форма объекта сложная, контур неровный или стороны минимального ограничивающего прямоугольника близки по размеру.

Возможные причины:

* сложная форма объекта;
* шумный контур;
* близкие значения ширины и высоты;
* чувствительность `minAreaRect` к форме контура.

## Ограничения

Текущая версия является программным прототипом.

Ограничения:

* результат зависит от качества YOLO-детекции;
* при пропуске объекта дальнейшая обработка невозможна;
* OpenCV-обработка чувствительна к освещению;
* контур может выделяться неточно при тенях и пересвете;
* угол ориентации может быть нестабильным для сложных форм;
* не учитываются физические размеры и ограничения реального захвата.

## Возможные улучшения

В дальнейшем проект можно улучшить:

* увеличить датасет;
* добавить изображения с разным освещением;
* улучшить параметры бинаризации и морфологии;
* добавить стабилизацию угла на серии кадров;
* протестировать алгоритм на видеопотоке;
* использовать сегментационную модель вместо обычной YOLO-детекции;
* связать результат с реальным манипулятором.

Наиболее перспективное улучшение - переход к сегментации, так как для объектов сложной формы маска объекта полезнее, чем обычный bounding box.
